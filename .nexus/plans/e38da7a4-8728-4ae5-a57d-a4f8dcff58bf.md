## FilterList: overflow menu + actions consolidation

### Why

Three related UX issues all touch the same set of files (`FilterListItem.tsx`,
`FilterDefinition`, the header action row, search visibility,
`ExcludeDropdown`/exclude-row, and the per-filter remove button). Because they
overlap on rendering and on the `FilterDefinition` type surface, they must
ship as a single PR — splitting would create inevitable merge churn and a
half-finished header API.

Today:

- The `...` overflow only appears for state types where `supportsExcluding`
  returns true (SELECT / EXACT_MATCH / CONTAINS_TEXT / TIMELINE). `linkedProperty`,
  NUMBER_RANGE, DATE_RANGE, TOGGLE, and HAS_LINK have no `...` at all, so
  users can't reach Keep/Exclude, Clear All, or Remove on those filters.
- The standalone remove `X` lives next to `...` and visually competes with
  the Keep/Exclude row the menu opens.
- MULTI_SELECT renders both the header search monocle *and* the `...`, but
  the inner Combobox already has search — the monocle is redundant noise
  with no opt-out.
- `FilterInputExcludeRow.handleToggleExclude` writes `isExcluding` to the
  *outer* state, which is wrong for `linkedProperty` (its `isExcluding` lives
  on the inner `linkedFilterState`).

### Approach

Item 1 (P0) — overflow menu on every filter:

1. Drop the `hasOverflowActions = supportsExcluding(...)` gate in
   `FilterListItem.tsx:138`. The `...` button always renders when there are
   actions to offer (which, with Remove living in the menu, is always when
   `onFilterRemoved` is provided or the filter has selections).
2. Replace `ExcludeDropdown` (Keep / Exclude only) with a content-aware menu
   that conditionally renders:
   - a Keep / Exclude toggle when `supportsExcluding(effectiveState)` is true,
   - a "Clear all selections" item when the filter has selected values,
   - a "Remove filter" item whenever `onFilterRemoved` is provided.
3. Delete the standalone remove `X` from the header. Removal becomes a menu
   action.
4. Compute the effective state once at the call site for linked filters:
   `effectiveState = filterState?.type === "linkedProperty"
     ? filterState.linkedFilterState : filterState`
   and pass it to `supportsExcluding` / selected-count helpers. No changes
   to the helpers themselves.
5. In `FilterInputExcludeRow.handleToggleExclude`, when the outer state is
   `linkedProperty`, toggle on the inner state and re-wrap before calling
   `onFilterStateChanged`. Read `isExcluding` and `selectedCount` from the
   inner state too.

Item 2 (P0) — MULTI_SELECT monocle opt-out:

1. Add `searchField?: boolean` (default `true`, backward compatible) to
   `FilterDefinition`. Document inline that `searchField: false` hides the
   header search monocle while keeping `...`.
2. Thread `definition.searchField` through `RenderFilterInput` /
   `FilterListItem` props so the gating happens in the header.
3. Gate the monocle:
   `showSearch = supportsSearch(effectiveState) && definition.searchField !== false`.

Item 3 (P1) — configurable filter action icons and placement:

1. Add a typed `actions` shape on `FilterDefinition`:
   `actions?: { search?: boolean | "menu"; overflow?: boolean; remove?: boolean | "menu"; placement?: "header-start" | "header-end" | "menu" }`.
2. Treat `searchField` (Item 2) as a narrower alias / shorthand for
   `actions.search`. If both are set, `actions.search` wins. Document the
   precedence.
3. Centralize action rendering in `FilterListItem` so search, overflow,
   clear-all, and remove share one placement decision (header vs. menu).
4. Keep all existing defaults backward-compatible: no consumer change means
   no visible change beyond the Item 1 + Item 2 deltas.

Cross-cutting:

- Update `FilterListItem.module.css` (`.headerActionButton`, exclude row
  layout) for the new menu / removed X.
- Update or add unit tests in `base/__tests__/` and
  `utils/__tests__/` covering: `...` rendering for every filter type,
  Remove from the menu, `searchField: false` hiding the monocle,
  `linkedProperty` Keep/Exclude toggling the inner state.
- Run `pnpm turbo typecheck --filter=@osdk/react-components` and
  `pnpm --dir packages/react-components vitest run` until green.
- Add a single changeset under `.changeset/` (patch — additive API,
  no breaking changes), bumping `@osdk/react-components`.
- If the public API surface changes (new `searchField` / `actions` fields
  on `FilterDefinition`), run
  `pnpm turbo check-api --filter=@osdk/react-components` and commit the
  updated `.api.md`.
- Run `npx dprint fmt` on changed files before committing.

### Out of scope

- Refactoring `ExcludeDropdown`'s internal styling beyond what the new menu
  contents require.
- Renaming `supportsExcluding` / `supportsSearch`. Spec says "no change to
  the helper itself" — call sites unwrap linked states; the helpers stay.
- Reworking the linked-property aggregation pipeline.
- Adding new filter component types.
