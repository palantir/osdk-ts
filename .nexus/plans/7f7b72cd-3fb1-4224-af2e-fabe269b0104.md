# FilterList: thread `renderValue` through linked-property inputs

## Why

`FilterListItemApi.ts` documents `renderValue?: (value: string) => ReactNode`
on the property filter definition and states:

> When the function returns a non-string ReactNode, search falls back to the
> raw value.

That contract is *partially* implemented today:

- The base inputs (`MultiSelectInput`, `SingleSelectInput`, `ListogramInput`)
  already accept `renderValue` and render JSX.
- `createRenderValueFilter` in `base/inputs/comboboxFilter.ts` already does the
  `typeof rendered !== "string"` fallback to the raw `itemValue` for search.
- The PROPERTY filter path (`PropertyFilterInput`) forwards
  `definition.renderValue` into all three base inputs.

The hole is the **LINKED_PROPERTY** path:

1. `LinkedPropertyFilterDefinition` (in `types/LinkedFilterTypes.ts`) does not
   expose a `renderValue` prop, so there is currently no way for a consumer to
   customize the chip/row/option rendering of a linked-property filter.
2. `LinkedPropertyInput.tsx` and its inner `LinkedMultiSelectInput`,
   `LinkedSingleSelectInput`, and `LinkedListogramInput` wrappers do not
   accept or forward `renderValue` to the base inputs.

The net effect: a consumer pointing a `LINKED_PROPERTY` filter at a string
column (e.g. an employee id on a linked object) cannot replace the raw id with
a friendly label the way they can on a top-level `PROPERTY` filter. There is
no current workaround short of forking the component.

## Approach

This is a faithful execution of the P0 spec — one worker task because every
edit lands in the same hot path (LinkedPropertyInput + its three inner
wrappers + the linked-property filter definition) and must ship atomically to
avoid a half-wired public API.

### Edits

1. **`types/LinkedFilterTypes.ts`** — add an optional `renderValue?: (value:
   string) => ReactNode` to `LinkedPropertyFilterDefinition`. Copy the JSDoc
   wording (and the "non-string ReactNode → search falls back to raw value"
   contract) from `FilterListItemApi.ts:298-305` so the public docs stay in
   sync. Add the `ReactNode` import from `react`.

2. **`inputs/LinkedPropertyInput.tsx`**:
   - Add `renderValue?: (value: string) => React.ReactNode` to
     `LinkedPropertyInputProps` and destructure it in `LinkedPropertyInputInner`.
   - Pull `renderValue` off `definition` (preferred, so callers configure it
     on the definition like PROPERTY does) and pass it to the relevant inner
     wrappers in the `MULTI_SELECT`, `SINGLE_SELECT`, and `LISTOGRAM` switch
     arms.
   - Extend `LinkedMultiSelectInputProps`, `LinkedSingleSelectInputProps`, and
     `LinkedListogramInputProps` with `renderValue?: (value: string) =>
     React.ReactNode` and forward it to `MultiSelectInput`, `SingleSelectInput`,
     and `ListogramInput` respectively.
   - Do **not** forward `renderValue` to the date / number / range / toggle /
     text-tags / contains-text arms — those base inputs don't accept
     `renderValue` today and the spec only covers the three select-style
     components.

3. **`FilterInput.tsx`** — `LinkedPropertyInput` already receives the full
   `definition`, so no change is needed here unless we choose to also accept
   `renderValue` as a direct prop on `LinkedPropertyInput`. Default plan: keep
   the API surface narrow — pull from the definition only, mirroring how
   `PropertyFilterInput` passes `definition.renderValue` downward.

4. **`base/inputs/comboboxFilter.ts`** — re-verify the ReactNode fallback is
   already present (it is) and add no further change. If the spec's "Suggested
   fix #2" is interpreted strictly, the worker can leave a brief code comment
   pointing at the existing fallback so future readers see the contract.

### Tests

5. **`base/inputs/__tests__/renderValue.test.tsx`** is already the canonical
   place for renderValue coverage; do **not** create a sibling
   `LinkedPropertyInput.renderValue.test.tsx`. Add a new
   `describe("LinkedPropertyInput renderValue", ...)` block at the bottom that:

   - Mounts `LinkedPropertyInput` for `MULTI_SELECT` (and at least one of
     `SINGLE_SELECT` / `LISTOGRAM` for breadth) with a stub `objectSet` /
     `definition` so the `usePropertyAggregation` hook returns deterministic
     data. Either mock the hook with `vi.mock` (preferred — keeps the test
     fast and OSDK-agnostic, matching `__tests__/LinkedPropertyInput.test.tsx`
     style), or hoist the existing mocks from that file.
   - Asserts that when `renderValue` returns JSX, the JSX appears in chips /
     listogram rows.
   - Asserts that `createRenderValueFilter`'s fallback covers search — either
     directly via the `createRenderValueFilter` unit, or by driving the
     listogram `searchQuery` with a raw substring and asserting only the
     intended row survives. The existing `ListogramInput renderValue
     (ReactNode)` block already proves the base behavior; the new test should
     prove the *linked* wrapper forwards the prop so the same search fallback
     reaches the underlying combobox.

   Follow `feedback_test_file_split.md`: append to the existing file, do not
   create a parallel `<feature>.test.tsx`.

### Public API + housekeeping

6. **`pnpm turbo check-api --filter=@osdk/react-components`** — the new
   `renderValue` field on `LinkedPropertyFilterDefinition` is a public API
   change. Commit the regenerated `etc/react-components.report.api.md` (or
   whichever report path the package uses).

7. **`.changeset/`** — add exactly one changeset (`patch` bump for
   `@osdk/react-components`) summarizing: "thread renderValue through
   LinkedPropertyFilter inputs". One file only; do not split.

8. **`npx dprint fmt`** on the changed files; do not bulk-format the repo.

9. **Verification loop** (per CLAUDE.md):
   - `pnpm turbo typecheck --filter=@osdk/react-components`
   - `pnpm --dir packages/react-components vitest run` (and `pkill -f vitest`
     afterwards)
   - `pnpm turbo check-api --filter=@osdk/react-components`
   - `pnpm --dir packages/react-components lint` (or `fix-lint` then re-run)

### Non-goals

- Threading `renderValue` into the date / number / range / toggle /
  text-tags / contains-text linked inputs (out of scope; their base inputs
  don't support it today).
- Refactoring `LinkedPropertyInput` to flatten the four sibling components,
  even though they share signatures — this PR is a tightly-scoped fix.
- Touching the `HAS_LINK` filter.
- Adding labels / i18n / inline-style theming knobs (per repo conventions).

### Risk

Public-API surface change on `LinkedPropertyFilterDefinition`, but it is an
**optional** field that mirrors the property-filter contract one-to-one, so
no existing consumer breaks. The API report bump must be committed alongside
the change so CI passes.
