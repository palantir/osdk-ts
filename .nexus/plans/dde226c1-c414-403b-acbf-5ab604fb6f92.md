## Why

The `FilterList` exposes a `formatDate?: (date: Date) => string` option on date/datetime property definitions that is supposed to control how dates are rendered in every date-typed filter input. Today it works end-to-end only for `DATE_RANGE` (and partially `MULTI_DATE` / `TIMELINE`):

- `DateRangeFilterInput` threads `formatDate` into histogram buckets, ticks, tooltip and the embedded `DatePicker`.
- `MultiDateFilterInput` → `MultiDateInput` forwards `formatDate` to chip labels but **not** to the underlying `DatePicker` used to pick new dates.
- `TimelineFilterInput` → `TimelineInput` forwards `formatDate` to the period labels above the inputs but the native `<input type="date">` value attribute stays ISO by design.
- `SingleDateFilterInput` does **not** accept `formatDate` at all. `PropertyFilterInput` does not forward `definition.formatDate` for the `SINGLE_DATE` case, `SingleDateInput` (base) doesn't accept the prop, and the underlying `DatePicker` (which already accepts `formatDate`) is therefore stuck on the default ISO `YYYY-MM-DD` idle text.

The net effect: a `FilterList` consumer who sets `formatDate` once on a datetime property sees consistent formatting on `DATE_RANGE` filters but inconsistent formatting on `SINGLE_DATE` / `MULTI_DATE` / `TIMELINE`. Within a single filter list, the same property can read as `Jan 15, 2024` in one place and `2024-01-15` in another. The `DateFormattingProps` JSDoc already claims `formatDate` controls "the multi-date chip text, and timeline labels" plus "DatePicker idle text" — so the runtime behaviour drifts from the documented contract for `SINGLE_DATE` and for `MULTI_DATE`'s picker.

A concurrent worker on the `filter-list-date-shortcuts` branch is adding a relative-range shortcuts rail to the same date inputs, so we want to establish a clean, uniform prop-threading pattern now that shortcuts work can mirror without further surface changes.

## Approach

Single worker task (the files overlap and must land together to keep filter formatting coherent):

1. **`SingleDateFilterInput`** — add `formatDate?: (date: Date) => string` prop and forward to `SingleDateInput`. No runtime default at this layer; pass through.
2. **`base/inputs/SingleDateInput`** — add `formatDate?: (date: Date) => string` prop and forward to the shared `DatePicker` (which already supports it). No default needed because `DatePicker` already falls back to `formatDateForInput` when undefined.
3. **`base/inputs/MultiDateInput`** — already accepts `formatDate` for chip labels; additionally forward it to the embedded `DatePicker` so the input field formats consistently with the chips.
4. **`base/inputs/TimelineInput`** — leave the JSDoc honest: HTML `<input type="date">` cannot be re-formatted, so `formatDate` continues to drive only the period labels. Verify the existing `formatDate` plumbing is correct; no other change.
5. **`FilterInput` / `PropertyFilterInput`** — forward `definition.formatDate` from `PropertyFilterDefinition` to `SingleDateFilterInput` in the `SINGLE_DATE` case (mirrors how `DATE_RANGE`, `MULTI_DATE`, `TIMELINE` already thread it).
6. **Docs** — refresh the `DateFormattingProps` JSDoc in `FilterListItemApi.ts` to make explicit that `formatDate` applies to all date-typed filter components: `DATE_RANGE`, `SINGLE_DATE`, `MULTI_DATE`, and `TIMELINE`. Call out the TimelineInput caveat (native HTML date input value remains ISO; only the period labels reformat) so consumers know what to expect.
7. **Tests**
   - Extend the existing `SingleDateInput.test.tsx` / `SingleDateFilterInput.test.tsx` (or append new `it` blocks within nested describes — do **not** split into `<X>.formatDate.test.tsx`) to cover: custom `formatDate` reaches the picker idle text; falling back to `formatDateForInput` when omitted.
   - Extend `MultiDateInput.test.tsx` (or the closest existing test) to assert the same `formatDate` value reaches the picker idle text in addition to chip text.
   - If no test file already exists for one of these, create a single base `Foo.test.tsx`, never `Foo.<feature>.test.tsx`.
8. **Verification**
   - `pnpm turbo typecheck --filter=@osdk/react-components`
   - `pnpm --dir packages/react-components vitest run`
   - `pnpm turbo check-api --filter=@osdk/react-components` and commit any `etc/*.api.md` diff (the public API of `FilterList` does not change because `formatDate` already lives on the definition, but base `SingleDateInput` is internal — confirm `.api.md` accordingly).
   - `npx dprint fmt` on changed files only.
   - One `.changeset/<name>.md` with `"@osdk/react-components": patch`.

## Out of scope

- `TimelineInput`'s native `<input type="date">` — re-formatting it would require a custom picker; deferred.
- The shortcuts rail (`filter-list-date-shortcuts` branch) — that work threads its own props through the same components; this task only standardises the existing `formatDate` plumbing so shortcuts can land cleanly on top.
- Any new `formatDate*` variants (datetime-specific, etc.). Keep the surface area unchanged.
