## Goal

Fix two visual alignment regressions in the FilterList histogram inputs:

1. **"No Value" row baseline misalignment.** In `NullValueWrapper.module.css`,
   `.nullValueRow` uses `align-items: center` while `.noValueLabel` inherits
   line-height and `.count` carries its own font tokens. The result is that
   the "No value" label and its count don't visually sit on the same text
   baseline as the histogram labels rendered above the row.

2. **Number-range hyphen drifts off vertical center.** In
   `RangeInput.module.css`, `.rangeInputs` has no explicit flex layout so the
   `.separator` hyphen is positioned via an intrinsic
   `calc(font-size + padding * 2)` height. When font metrics differ between
   the inputs and the separator span, the hyphen floats above or below the
   actual input center instead of being flex-centered against the real input
   boxes.

## Why one worker, not two

Both issues live in the same `base/inputs/` folder, both are pure CSS edits
to module files that ship together, and they share the same review surface
(changeset, formatting, typecheck). The original spec explicitly directs the
planner to keep this as a single task â€” splitting would create two parallel
PRs touching adjacent files in the same component family with no
independent value.

## Approach

### 1. `NullValueWrapper.module.css`

Change `.nullValueRow` from `align-items: center` to `align-items: baseline`
so the label text and the count number sit on the same typographic
baseline, matching the histogram count labels above. (Per the spec, the
alternative â€” pinning an explicit `line-height` token on `.noValueLabel` â€”
is acceptable, but the baseline switch is the lighter-touch fix and is
what the spec lists first.)

### 2. `RangeInput.module.css`

- Promote `.rangeInputs` to a proper flex row that centers the hyphen
  against the inputs: keep `display: flex`, change
  `align-items: flex-end` â†’ `align-items: center`, and keep the existing
  `gap` (consider switching to a dedicated
  `--osdk-filter-range-separator-gap` token if one exists in `base.css`,
  otherwise leave the current `--osdk-filter-range-inputs-gap` in place
  rather than introducing an undefined token â€” flag as a separate token
  follow-up if needed).
- Drop the intrinsic `height: calc(...)` from `.separator` so it no longer
  fights flex centering. Keep `display: flex; align-items: center;` (or
  reduce to whatever is needed once flex on the parent does the work).
- Because `.rangeInputs` now centers everything (not `flex-end`), the
  `.inputWrapper` columns â€” which stack a label above an input â€” will need
  to remain visually consistent. The current layout uses `flex-end` so the
  inputs themselves align along the bottom row regardless of label height.
  The worker must verify that flipping to `align-items: center` doesn't
  regress label/input stacking; if it does, the right fix is to leave the
  parent at `flex-end` and instead align ONLY the separator (e.g. give
  `.separator` `align-self: center` and an explicit `height: 100%` matched
  via flex, or move the separator into its own centered wrapper). Pick
  whichever produces the cleanest result with the inputs visually aligned
  and the hyphen centered on the input row.

- If a token is genuinely missing (e.g. a separator-gap token referenced
