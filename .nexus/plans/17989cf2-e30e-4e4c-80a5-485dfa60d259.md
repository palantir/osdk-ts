# Plan: FilterList — opt-in relative-range shortcuts in date picker

## Why

Today's calendar popover in `@osdk/react-components` exposes a single
shortcut: a "Today" button rendered by `DateCalendar`. Users filtering
on `datetime`/`timestamp` properties commonly want quick relative
ranges (past hour, past day, past week, past month, etc.) without
clicking through the calendar. There is currently no helper for those
periods in `shared/dateUtils.ts`, and the `DATE_RANGE` filter state
stores absolute `Date`s with no notion of a relative period.

This goal adds an **opt-in** shortcuts rail to the date filter inputs.
Default behavior is unchanged — consumers must explicitly enable
`dateShortcuts` on the `FilterDefinition` for shortcuts to appear.

## Approach

This is a single self-contained, opt-in additive change. All edited
files live under `packages/react-components/src/filter-list/` plus
`shared/dateUtils.ts`. The work touches overlapping prop-threading
paths, so it MUST land as one PR.

### 1. Shared date utility — relative periods

In `packages/react-components/src/shared/dateUtils.ts`:

- Export a string-literal union
  ```ts
  export type RelativeDatePeriod =
    | "past-hour"
    | "past-day"
    | "past-week"
    | "past-month"
    | "past-3-months"
    | "past-6-months"
    | "past-year"
    | "past-2-years";
  ```
- Export
  ```ts
  export function getRelativeDateRange(
    period: RelativeDatePeriod,
    now?: Date,
  ): { min: Date; max: Date };
  ```
  Returns `{ min: now − period, max: now }`. Use a `now` parameter
  (defaulting to `new Date()`) so unit tests can pin the clock.
  Calendar-style periods (`past-month`, `past-3-months`, `past-6-months`,
  `past-year`, `past-2-years`) subtract via `setMonth` /
  `setFullYear` rather than naive ms math so DST + month-length
  edge cases stay correct. Hour / day / week subtract via ms.
- Export the canonical default list (in the order from the spec):
  ```ts
  export const DEFAULT_RELATIVE_DATE_PERIODS: readonly RelativeDatePeriod[] = [
    "past-hour", "past-day", "past-week", "past-month",
    "past-3-months", "past-6-months", "past-year", "past-2-years",
  ];
  ```
- Export a label resolver:
  ```ts
  export function getRelativeDatePeriodLabel(p: RelativeDatePeriod): string;
  ```
  English only (per repo convention — no i18n).
- Add unit tests in `shared/__tests__/dateUtils.test.ts` (append to
  existing `describe`s; do NOT split into a new file). Cover at least:
  hour math, day math, month math (Mar 31 → Feb 28/29 doesn't roll
  over), DST boundary, and default list ordering.

### 2. API surface — opt-in prop on the filter definition

In `packages/react-components/src/filter-list/FilterListItemApi.ts`:

- Add `RelativeDatePeriod` to the re-exports (or import from
  `../shared/dateUtils.js` and re-export from the public surface).
- Extend `DateFormattingProps` (or add a sibling `DateShortcutsProps`
  interface — prefer the latter for readability) with:
  ```ts
  /**
   * Opt-in relative-range shortcuts rendered as a rail inside the
   * date filter popovers.
   *
   * - `undefined` (default): no shortcuts rendered. No behavior change
   *   from prior releases.
   * - `true`: render the full default list
   *   ({@link DEFAULT_RELATIVE_DATE_PERIODS}).
   * - `RelativeDatePeriod[]`: render exactly these periods in the
   *   given order.
   *
   * Clicking a shortcut emits an absolute `{ min, max }` Date range —
   * the picker's underlying `FilterState` stays absolute. Sticky
   * relative-label persistence (remembering "Past 3 months" across
   * refreshes) is intentionally deferred; that would require a new
   * `FilterState` variant.
   */
  dateShortcuts?: RelativeDatePeriod[] | boolean;
  ```
- Wire it through `PropertyFilterDateExtras<P>` so it is allowed only
  on `datetime` / `timestamp` properties and `never` otherwise —
  mirroring the existing `formatDate` mechanism.

### 3. Base UI primitive — `ShortcutBar`

New file:
`packages/react-components/src/filter-list/base/inputs/ShortcutBar.tsx`
with a sibling `ShortcutBar.module.css`.

Props (pure / OSDK-agnostic, suitable for direct reuse from the base
components):
```ts
interface ShortcutBarProps {
  periods: readonly RelativeDatePeriod[];
  onSelect: (range: { min: Date; max: Date; period: RelativeDatePeriod }) => void;
  className?: string;
}
```

Render a vertical (left-rail) list of `ActionButton` (from
`base-components/action-button/`) using the resolver from step 1 for
labels. CSS only via `--osdk-*` tokens — read `src/tokens/base-tokens/
base.css` first; if a needed token doesn't exist, flag it as a
follow-up rather than inlining a value.

Helper to resolve the `dateShortcuts` prop to a `readonly
RelativeDatePeriod[] | undefined`:
```ts
export function resolveDateShortcuts(
  prop: RelativeDatePeriod[] | boolean | undefined,
): readonly RelativeDatePeriod[] | undefined;
```
Co-locate it next to `ShortcutBar.tsx` (or in `dateUtils.ts` — pick
whichever keeps the dependency graph one-way).

### 4. Render the shortcut bar in each date input

The bar appears as a left rail beside the calendar (or beside the
range histogram), only when `periods != null`.

Thread `dateShortcuts` through:

- `inputs/DateRangeFilterInput.tsx` → `base/inputs/
  DateRangeHistogramInput.tsx` (wrap the existing `RangeInput` with a
  flex row that renders `<ShortcutBar/>` on the left when shortcuts
  resolve to a non-empty list). When clicked, call `onChange(min, max)`.
- `inputs/SingleDateFilterInput.tsx` → `base/inputs/SingleDateInput.tsx`
  → render the bar to the left of the picker. On click, set the
  selected date to `max` (the present moment). The spec calls for a
  "range" click; for `SINGLE_DATE`, treat the click as "snap to now."
  (Document this decision in the prop JSDoc on `FilterListItemApi`.)
- `inputs/TimelineFilterInput.tsx` → `base/inputs/TimelineInput.tsx` →
  render the bar inside the `.timeline` container, before the
  `.labels` row. On click, call `onChange(min, max)`.

The exact mounting strategy for the `DatePicker` popover used by
`SingleDateInput` is: the `DatePicker` itself does not gain new props
in this change. Instead, render the `ShortcutBar` *outside* the
`DatePicker` (as a sibling in the row laid out by `SingleDateInput`).
This avoids changes to the public `DatePicker` surface and keeps the
calendar untouched. (If, during implementation, the worker finds that
the shortcuts must live inside the popover to feel right visually,
they may add an optional `shortcuts?: ReactNode` slot prop to
`DatePicker` and pass the rendered bar as the slot — that keeps the
shared picker generic and avoids leaking filter-list concepts into
`shared/calendar/`.)

### 7. Storybook (best-effort)

If a `FilterList` story already exists for date-typed properties,
extend it with a knob/story variant that sets `dateShortcuts: true`
so reviewers can see the rail. If no story exists for date filters,
do NOT speculatively add one — flag as a follow-up.

## Risks / known interactions

- A concurrent worker on `filter-list-format-date-inputs` is adding
  `formatDate` to `SingleDate` / `Multi` / `Timeline` inputs. They
  thread props through the same components. The plan above already
  follows their pattern. If their branch lands first, rebase and
  re-thread `dateShortcuts` alongside their `formatDate` plumbing —
  the two props are independent.
- Sticky relative-label persistence requires a new `FilterState`
  variant and is explicitly deferred. Do NOT add it here.

## Non-goals

- No changes to `DatePicker`'s public props if avoidable (see step 4).
- No i18n / `labels` prop on the bar — English only per repo convention.
- No multi-select / "compare to" shortcuts.
- No keyboard chord (e.g. "press M for past-month") — buttons only.
